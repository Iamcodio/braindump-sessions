## BUSINESS/TECHNICAL DOMAIN

### BREAKTHROUGH: "We're Outsourcing Overwhelm"
- Three-word product positioning that crystallizes entire iamcodio methodology
- Not managing (implies you keep it), not reducing (implies you fight it), not coping (implies you tolerate it)
- OUTSOURCING it - like payroll, infrastructure, any business process
- What gets outsourced: Ideation loop -> brain dump, memory -> database, decisions -> pile system, patterns -> vector search, context -> Claude
- User keeps creative engine, offloads overhead
- This IS the business model: Outsourcing cognitive overhead for generative minds

### Card-Based Knowledge System Architecture (Refinement)
- Problem identified: Claude Desktop projects burn 60-70% context just loading project knowledge
- With 30-40 chats containing artifacts, context-starved before meaningful work begins
- Solution: Atomic card system (tweet-sized, 500 chars max) with pile-based workflow
- Historical note: Originally conceptualized months ago from Pirsig's Lila index card method
- Pile names changed from Pirsig's originals (copyright - Pirsig Estate) to Shape Up terminology

### Six-Pile System (Pirsig Method + Shape Up)
1. INBOX - Raw brain dump cards, unprocessed (was UNASSIMILATED)
2. SHAPING - Cards being formed into pitches (was PROGRAM)
3. ACTIVE - Currently building in 6-week cycle (was CATEGORIZED working)
4. SHIPPED - Completed, archived (was CATEGORIZED done)
5. HOLD - Future consideration (was GRIT/TOUGH)
6. KILL - Discard pile (was JUNK)

### Context Efficiency via Pile Loading
- Claude loads ONLY relevant pile (e.g., ACTIVE = 5-20 cards, ~7,500 chars)
- INBOX reviewed periodically, not auto-loaded
- SHIPPED searchable via vector DB, loaded on-demand
- HOLD/KILL excluded from active context
- Result: 30% context vs 60-70% in current system
- Scales indefinitely without context bloat

### Database Architecture (Hybrid)
- PostgreSQL: Card metadata, pile assignment, relationships, temporal data
- pgvector: Semantic embeddings for similarity search
- Redis: Job orchestration for async processing
- Pile movement history table for audit trail
- Query patterns: Semantic, thread following, temporal, tag-based, pile-based

### Svelte Interface Design
- Kanban-style pile view with drag-and-drop
- Visual columns for each pile
- Card detail modal for editing
- Search highlights across all piles
- Pile filtering for focused work
- Pile count badges (e.g., "INBOX: 47 cards")
- Color-coded by pile for visual hierarchy

### Shape Up Integration
- Brain dump -> INBOX (raw ideas)
- Shaping session -> SHAPING (forming pitches)
- Betting table -> ACTIVE (6-week build)
- Cycle complete -> SHIPPED (archived)
- Not ready -> HOLD (revisit later)
- Wrong direction -> KILL (learn from failures)

### Implementation Stack
- PostgreSQL with pgvector extension
- FastAPI for query interface
- Python sentence-transformers for embeddings
- Svelte/Tauri for frontend (optional, CLI viable)
- Filesystem MCP integration for brain dump access
- SuperWhisper transcription -> auto-card generation
